# PWA & Service Worker

The app is installable on desktop and mobile browsers as a Progressive Web
App, which is also the foundation [[Mobile-Packaging|Capacitor]] builds on
top of for native Android/iOS packaging.

## Manifest

`manifest.webmanifest` (repo root, copied into `www/` by
[[Build-Pipeline|build_www.py]]):

- `name`: "Road Trip Weather", `short_name`: "TripWeather"
- `display: "standalone"` â€” launches without browser chrome
- `orientation: "portrait"`
- `background_color` / `theme_color`: `#0f172a` (matches the dark theme)
- Icons: 192px, 512px, and a 512px `maskable` variant (for adaptive icons on
  Android), generated by `make_icons.py` into `static/icons/`

## App shell meta tags

`templates/index.html` (lines 4â€“15) sets:
- `theme-color`, `viewport-fit=cover` (for notch/safe-area handling)
- `mobile-web-app-capable` (Android/Chrome) and `apple-mobile-web-app-capable`

- `apple-touch-icon` pointing at the 180px icon (iOS home-screen icon size)

## Service worker (`sw.js`)

66 lines. Registered non-fatally at the bottom of `index.html` â€” if
registration fails, the app still works, just without offline support.

**Cache strategy is a hard split by URL, not a generic runtime cache:**

- **Cache-first, versioned shell** (`SHELL` array, `sw.js:11-23`): the HTML
  shell (`.`), manifest, `static/style.css`, `static/api.js`,
  `static/app.js`, all four icon files, and the Leaflet CSS/JS from the
  `unpkg.com` CDN. Precached on `install`. Cache name is
  `rtw-shell-v7` â€” **bump this version string whenever shell assets change**,
  since that's what triggers the old cache to be deleted on `activate`
  (`sw.js:31-37`) and forces clients to pick up the new files.
- **Network-only, always** (`sw.js:43-49`): any request whose hostname ends
  in `nominatim.openstreetmap.org`, `photon.komoot.io`,
  `router.project-osrm.org`, `api.open-meteo.com`, or
  `tile.openstreetmap.org`. These are explicitly excluded from caching â€”
  forecasts must be fresh, and tiles would blow well past any reasonable
  cache budget.
- Everything else falls through to cache-first-with-network-fallback
  (`sw.js:53-65`), and same-origin or `unpkg.com` responses get opportunistically
  cached on the way through.

**Practical implication:** the installed app opens instantly and its UI
shell works fully offline, but planning any trip still requires a network
connection â€” there is no offline trip-planning mode.

See also [[Limitations-and-Risks]] for the iOS cache-eviction caveat
documented in `MOBILE.md`.
